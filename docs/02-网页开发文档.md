### 1.1.2 MVP 范围标注（客户《功能清单》四、简化版 MVP - 网页）

> 四-网页 MVP 清单 = 后台登录 / 项目管理 / 巡查记录查看 / 地图查看巡查轨迹 / 地图查看问题点位 / 查看问题照片 / 问题列表管理 / 数据统计 / **Excel 导出**。映射到本文 Phase：

- **MVP 必交付 Phase**：P0（后台登录外壳）、P3（项目管理 + 巡查记录查看）、P4（地图查看轨迹 + 问题点位）、P5（查看问题照片/全景）、P6（问题列表管理）、P7（数据统计）、P8（Excel 导出）。
- **MVP 横切支撑（保持轻量但必需）**：P1（字典配置 —— 问题类型/状态/分类是问题管理与统计的前置数据）、P2（RBAC —— 「后台登录」必须有鉴权，最小化即可）。
- **MVP 导出范围 = 三种 Excel 全含**：`INSPECTION_RECORDS` + `PROBLEM_LIST` + `PROJECT_STATS`。因 MVP 网页清单的「数据导出」对应客户三-8 的三项导出，**`PROJECT_STATS`（项目统计数据 Excel）属于 MVP 范围，不可省略**。
- **明确不在 v1 范围**：实时预览 360 画面（无直播预览）、录像、HDR-only 流程 —— 这些是 App/拍摄侧的范畴且已被裁出 v1，Web 端无对应功能。

### 1.2 不在范围内

- 后端实现（OpenAPI spec、Go 服务、SQL）—— 见后端开发文档。
- App 端（Kotlin / Insta360 SDK / 离线 outbox）—— 见 App 开发文档。
- **坐标系持久化**：Web 永远从后端读写 WGS84/SRID 4326，仅在腾讯地图渲染边界做 WGS84↔GCJ-02 转换（见 §4.3）。
- **.insp 原始鱼眼拼接**：在 App 端完成（`ExportUtils.exportImage`），Web 只渲染后端给出的 equirectangular JPG。

读者应熟悉：React 19 函数式组件 + Hooks、TypeScript 严格模式、TanStack Query 缓存模型、Ant Design v5 + Pro Components、地图投影坐标系基础概念（WGS84 / GCJ-02 偏移）。

## 3. 总体架构 / 模块划分

### 3.1 分层架构

```
┌──────────────────────────────────────────────────────────┐
│  Pages (路由页面: 按 feature 组织)                          │
│   projects / inspections / problems / dict / rbac / stats  │
├──────────────────────────────────────────────────────────┤
│  Features (领域逻辑: hooks + 组件 + service)                │
│   每个 feature 自含 api(query hooks) / components / types   │
├──────────────────────────────────────────────────────────┤
│  Shared 基础设施                                            │
│   apiClient(axios+envelope+jwt) │ generated openapi types  │
│   queryClient(TanStack) │ authStore/uiStore(zustand)       │
│   geo(WGS84↔GCJ02) │ map(腾讯GL封装) │ psv(全景封装)        │
│   i18n │ dayjs(tz) │ guards(路由权限) │ errorBoundary       │
└──────────────────────────────────────────────────────────┘
```

**关键原则（遵循全局 coding-style）**：
- 多个小文件 > 少数大文件；单文件 200–400 行，硬上限 800 行。
- 按 feature/domain 组织，不按文件类型组织。
- 不可变更新（构造新对象，不就地修改）—— Zustand store 用展开返回新 state。
- 边界做输入校验（zod 校验表单、环境变量、后端返回的 JSON 配置块）。
- 错误显式处理，UI 面向用户给友好提示，控制台 / 上报留详细上下文，绝不静默吞错。

## 4. 与其他端的契约引用

> Web 是后端契约的纯消费者。以下表名 / 接口名 **必须逐字使用**，与「数据字典」「API 契约」一致。

### 4.1 数据模型引用（共享表名）

Web 渲染 / 写入对应以下后端表（仅经 API，不直连 DB）：

- 身份 / RBAC：`users`、`roles`、`permissions`、`user_roles`、`casbin_rule`
- 字典 / 配置：`dict_type`（含 `version` / `content_hash` = ETag）、`dict_item`（`is_active=false` 为「退役」非删除）、`app_config`（`export.image`、`capture.default`，`content_hash` = ETag）
- 项目 / 任务：`projects`（`custom_fields` jsonb 由 `dict_scope='project_field'` 驱动；`area_geom` MultiPolygon 4326）、`inspection_tasks`
- 巡查 / 轨迹：`inspections`（`duration_seconds`、`mileage_meters` 为 `ST_Length(route_geom::geography)` 米、`point_count`、`route_geom` LineString 4326）、`trajectory_points`（高频 GPS，`geom` Point 4326）
- 问题 / 处理：`problems`（`type_item_id`/`status_item_id`/`category_item_id` 软 FK→`dict_item`、`dict_version_used` 锁版本、`cover_media_id` 冗余封面、`geom` Point 4326）、`problem_processing_log`（追加式审计：`action` ∈ `STATUS_CHANGE|COMMENT|REASSIGN`，仅此三种；其中 `STATUS_CHANGE` **仅由后端在状态变更事务内写入**，前端只 POST `COMMENT`/`REASSIGN`，见 §4.5 D3）
- 媒体：`media_assets`（三层 `tier` = `original|web|thumb`；`capture_state` 状态机 `CAPTURED_RAW→STITCHED→QUEUED→UPLOADING→UPLOADED→CONFIRMED`；`verified_at` 仅在后端 HeadObject 校验后置位；`media_group` 关联同一逻辑照片三层）
- 导出：`export_jobs`（`job_uuid` 公开句柄；`status` ∈ `PENDING|RUNNING|SUCCEEDED|FAILED|CANCELLED`；`progress` 0..100；`result_cos_key`）

> **Web 几乎不创建 `media_assets`**（媒体由 App 离线采集 → STS 多段上传 → 后端 HeadObject 确认）。Web 只 **读取**：`GET /api/v1/media/{id}` 返回按 tier 的 **签名 CDN URL**。全景查看用 `tier=web` 的 equirectangular JPG，列表缩略图用 `tier=thumb`。

### 4.2 API 契约引用

**响应信封（所有 JSON 响应）**：
```jsonc
{ "success": true,  "data": <T|null>, "error": null, "meta": null }
{ "success": false, "data": null, "error": {"code":"VALIDATION_FAILED","message":"...","details":[...]}, "meta": null }
```
列表端点 `meta` = `{ total, page, page_size }`。HTTP 状态镜像信封（200/201；401/403/404/409/422/429/500，详见 §4.6 错误码契约）。

**axios 拦截器职责（`shared/api/http.ts`）**：
1. 请求拦截：附加 `Authorization: Bearer <access_token>`。
2. 响应拦截：成功时 **拆封返回 `data`**（业务层拿到的是 `T` 不是信封）；`success=false` 抛 `ApiError(code, message, details, httpStatus)`。
3. 401 → 尝试 `POST /api/v1/auth/refresh`（轮换 refresh），**单飞（single-flight）合并并发刷新**；刷新失败 → 清 store + 跳登录。


### 4.3 坐标系契约（WGS84 / GCJ-02）—— 强约束

- **后端永远 WGS84/SRID 4326**：`/inspections/{id}/trajectory`、`/problems/map`、`/problems/{id}` 的 `geom`、`route_geom`、`area_geom` 全是 WGS84。
- **GCJ-02 绝不持久化**。仅在「腾讯地图渲染边界」做转换：
  - 读：后端 WGS84 → `wgs84ToGcj02()` → 传给腾讯地图绘制。
  - 写（地图取点 / 拖拽，例如管理端微调问题点）：腾讯地图 GCJ-02 → `gcj02ToWgs84()` → 提交后端。
- **唯一转换工具**：`shared/geo/coordTransform.ts`，**必须有单测**（往返误差 < 1e-6 度、典型城市点位偏移方向正确）。任何组件都不得内联坐标转换。
- 注意腾讯地图 GL 的 `LatLng(lat, lng)` 入参顺序与 GeoJSON `[lng, lat]` 相反 —— 在 map 封装层统一转换，避免散落各处。

### 4.4 同步契约（仅了解，Web 不写）

`POST /api/v1/sync/batch` 由 App 端用于离线幂等批量 upsert（`client_uuid` UNIQUE + `INSERT ... ON CONFLICT` + 逐项 SAVEPOINT；逐项 accepted/rejected/duplicate）。**Web 端不调用此接口**，但需理解：列表中出现的 `inspections`/`problems`/`media_assets` 可能来自离线回传，**可能携带已退役字典项**（`dict_version_used` 旧版本）。**Web 渲染问题类型 / 状态标签时，必须能渲染 `is_active=false` 的历史字典项**（见 P1 / P6 实现要点）。

### 4.6 错误码契约（与契约 00 错误码目录逐字一致）

> 信封 `error.code` 为以下闭集；`error.details[]` 元素形如 `{ field, code, message }`。`shared/api/apiError.ts` 的 `ApiErrorCode` 联合类型必须与此表完全一致。

| `error.code` | HTTP | Web 端处理 |
|---|---|---|
| `VALIDATION_FAILED` | 422 | 表单字段级回填 `error.details[].field`，定位到对应表单项 |
| `UNAUTHENTICATED` | 401 | 缺失/非法 access token → 清 store 跳登录（**注意：不是 `UNAUTHORIZED`**） |
| `TOKEN_EXPIRED` | 401 | access JWT 过期 → 触发 `/auth/refresh` 单飞刷新并透明重放（与 `UNAUTHENTICATED` 分支区分） |
| `FORBIDDEN` | 403 | casbin 拒绝 → 即使 UI 已隐藏控件也必须友好提示（前端权限仅 UI 层） |
| `NOT_FOUND` | 404 | 资源不存在或已软删 → 空态/返回列表 |
| `CONFLICT` | 409 | 唯一冲突/删除受限（如有巡查的项目）/状态机非法 → 友好提示 + 引导刷新 |
| `MEDIA_VERIFY_FAILED` | 409 | `/media/confirm` HeadObject 不一致（Web 几乎不触发；只读场景一般无） |
| `DICT_VERSION_RETIRED` | 409 | **保留码**：历史容忍同步不触发它；Web 仅需在 `ApiErrorCode` 中登记 |
| `RATE_LIMITED` | 429 | 热点接口限流（登录等）→ 退避重试 + 提示稍后再试 |
| `INTERNAL` | 500 | 兜底错误 → 通用「服务异常」提示，不泄露后端细节 |

> **`http.ts` 401 分支**：必须先看 `error.code`：`TOKEN_EXPIRED` → 走单飞 `refresh` 重放；`UNAUTHENTICATED` → 直接登出。两者均为 401，不能仅凭状态码判定。

**关键实现要点与坑**：
- **信封拆封一次到位**：响应拦截器返回 `resp.data.data`，业务 hook 拿到的就是 `T`；`meta`（分页）需另存（封装 `fetchList` 返回 `{ items, total, page, pageSize }`）。
- **401 单飞刷新**：并发请求同时 401 时，只发一次 `refresh`，其余请求挂起等待新 token 后重放；刷新接口本身 401 → 直接登出，避免无限循环。**`refresh` 请求不能再走会触发刷新的拦截器**（用裸 axios 或标记跳过）。
- **`/auth/me` 的 permission codes 是动态菜单 + 路由守卫的唯一权威**，不要在前端硬编码角色判断（casbin 在后端，前端仅做 UI 隐藏，**真正鉴权在后端**，403 仍可能返回 → 必须处理）。
- token 存 `localStorage` 便于刷新页面保活；**绝不**把密码 / refresh 明文打日志。
- 时间：后端均 UTC `timestamptz`，**渲染统一走 `dayjs.tz(ts,'Asia/Shanghai')`**，避免各页面各自 `new Date()`。

**验收标准（DoD）**：
- 登录 → 拿 token → `/auth/me` → 菜单按权限渲染 → 刷新页面保持登录 → 改密 → 登出全链路通。
- access 过期（`error.code=TOKEN_EXPIRED`）触发 401 → 自动 refresh → 原请求透明重放成功（开发者工具可见仅一次 refresh）；`UNAUTHENTICATED` 则直接登出（不刷新）。
- 无权限用户访问受限路由跳 403 页；菜单不出现无权限项。
- TS 严格模式零 `any`（生成类型除外）；ESLint / Prettier 零错误。

**测试要求**：
- 单元：`http.ts` 拦截器（信封拆封、`TOKEN_EXPIRED` 单飞刷新与透明重放、`UNAUTHENTICATED` 直接登出、refresh 失败登出）用 msw 模拟；`authStore` reducer 不可变性。
- 组件：`LoginPage` 表单校验 + 提交（msw）。
- E2E：登录 → 改密 → 登出（Playwright，msw 或后端 dev）。

**关键实现要点与坑**：
- **ETag 拉取（强约束）**：`GET /dict/types/{code}/items` 返回 `ETag: "<dict_type.content_hash>"`，客户端 `If-None-Match` 命中返回 304。**axios 默认对 304 抛错** —— 需配置 `validateStatus` 允许 304 或在拦截器特判，命中走本地缓存。响应体含 `version` + `content_hash` 以便缓存更新。
- **历史字典版本容忍（强约束）**：字典缓存 **必须保留 `is_active=false`（已退役）项**，因为离线回传的问题可能引用退役类型（`dict_version_used` 旧版本）。`DictTag` 找不到 active 项时回退到退役项渲染，**绝不显示「未知」或报错**。
- **配置块是有版本历史的**：`app_config` 每个 `config_key` 只有一个 `is_active=true` 当前版本；PUT 产生新版本（后端处理）。前端展示 `content_hash` 作为 ETag，提交前可乐观锁（带上当前版本，冲突 409 提示刷新）。
- `extra` jsonb 编辑器要 zod 校验结构，**永不信任**手填 JSON（边界校验原则）。

**验收标准（DoD）**：
- 新增 / 编辑 / 停用问题类型、状态、分类、拍照预设全部可用。
- 修改任一 `dict_item` 后，`dict_type.version` 与 `content_hash` 变化 → 前端字典缓存按 ETag 自动失效刷新。
- 已停用的字典项在历史数据中仍正常渲染（带「已停用」标识）。
- `export.image` / `capture.default` 可编辑并可见历史。

**测试要求**：
- 单元：`useDict` ETag 流程（304 命中缓存、200 刷新）；`dictCache` 不可变更新；`DictTag` 渲染退役项。
- 组件：`DictItemTable` 停用/启用切换；`ConfigPage` zod 校验拒绝非法 jsonb。
- E2E：新建问题类型 → 在问题管理下拉出现 → 停用 → 历史问题仍显示该类型标签。

**测试要求**：

  - 确认图为 **2:1 等距柱状（equirectangular）**，球面无撕裂 / 无地平线错位。
  - 检查图是否带 **GPano XMP** 元数据；**决策**：是否需要后端/前端依赖 GPano。**结论应为：PSV 的 equirectangular adapter 不依赖 GPano，只要图是 2:1 等距柱状即可正确渲染**（X3 导出已是该格式）；记录验证结论到本仓库 README。
  - 若发现非 2:1 或上下黑边，反馈 App 团队调整 `ExportUtils.exportImage` 的 `ExportMode.PANORAMA` 尺寸/比例。

**关键实现要点与坑**：
- **Web 无 Insta360 SDK（强约束）**：Web 端 **不做任何拼接**。`.insp` 双鱼眼原始图的拼接在 App 端用 `ExportUtils.exportImage(ExportMode.PANORAMA, 2:1)` 完成，产出降采样压缩的 equirectangular JPG。Web 只用 PSV 渲染该 JPG。
- **只渲染已确认媒体**：只显示 `verified_at IS NOT NULL`（`capture_state=CONFIRMED`）的媒体；未确认的不显示（避免渲染未完成上传的对象）。
- **tier 选择（D4）**：App **只上传 `tier=original`**，后端 worker 在 `original` 达 `CONFIRMED` 后**派生 `web`+`thumb`**（同 `media_group` 的兄弟 `media_assets` 行）。Web 全景查看用 `tier=web`（管理员可配尺寸/质量，见 P1 `export.image`），缩略图导航用 `tier=thumb`，`original` 仅按需 / 下载用。Web 不参与任何 tier 派生。
- **签名 CDN URL 有时效**：`GET /media/{id}` 返回的是腾讯 CDN 签名 URL，过期需重新获取。PSV 加载失败的错误回调里重拉。
- **GPano 决策（D5）**：PSV v5 equirectangular adapter **不强制要求** GPano XMP，只要图是 2:1 等距柱状即可渲染球体。但按 D5 跨端约定，**App 端仍会写入 GPano XMP（8 字段模板）以保证互操作/兜底**；Web 不解析也不依赖 XMP，仅以「2:1 等距柱状」为渲染前提。验证后明确记录，避免误加无谓依赖。
- 卸载销毁 PSV 实例防 WebGL 上下文泄漏。

**验收标准（DoD）**：
- 真实 X3 导出 JPG 在 PSV 中作为完整球体正确渲染（无撕裂、地平线水平、可拖拽/缩放）。

- 签名 URL 过期能自动重拉重试。

**测试要求**：
- 单元：`media_group` → tier URL 选择逻辑；签名过期重拉逻辑。
- 组件：`PanoramaViewer` 用 mock PSV 验证以正确 URL 初始化、卸载调 `destroy`。
- 手工/真机：真实 X3 导出图渲染人工验收（截图存档）。
- E2E：地图问题点「看全景」→ 弹层 → PSV 容器渲染（Playwright 断言 canvas 出现）。

**涉及模块/接口/表**：`features/problems/*`；接口 `/problems` `/problems/{id}` `/problems/{id}/logs`；表 `problems`（软 FK `type_item_id`/`status_item_id`/`category_item_id`、`dict_version_used`、`cover_media_id`）、`problem_processing_log`（追加式）。

**关键实现要点与坑**：
- **字典软 FK + 历史版本（强约束）**：`type_item_id`/`status_item_id`/`category_item_id` 是软 FK→`dict_item`（后端 `ON DELETE RESTRICT`，只能退役不能硬删）。渲染标签用 P1 的 `DictTag`，**必须能渲染已退役（`is_active=false`）的历史项**（离线回传问题可能用旧字典版本，`dict_version_used` 记录捕获时版本）。**筛选下拉用 active 项，但列表渲染允许退役项**。
- **处理日志追加式（强约束）**：`problem_processing_log` 是 **insert-only、无 `updated_at`、无软删**。UI 只「追加」，不提供编辑 / 删除日志。
- **`captured_at` 是拍照时间**（设备端），渲染走 Asia/Shanghai。
- **封面**：`cover_media_id` 是冗余封面（denormalized），列表直接用它取 `thumb` 签名 URL，避免 N+1。
- **状态变更日志由后端原子写（D3，强约束）**：`PUT /problems/{id}` 改 `status_item_id` 时，后端在同一 DB 事务内追加 `STATUS_CHANGE` 日志。前端**只发一次 PUT**，**绝不** POST `STATUS_CHANGE`，无需自行保证「PUT + POST」原子性。仅 `COMMENT`/`REASSIGN` 由前端 POST。
- 问题点位地图（详情内小地图）复用 P4 的 `coordTransform` + map 封装。

**验收标准（DoD）**：
- 问题列表按 6 个维度筛选；封面 / 类型 / 状态标签正确（含退役项）。
- 详情可看全景、看点位、改分类/状态/备注、看完整处理时间线。
- 改状态（单次 PUT）后，处理时间线自动出现后端生成的不可篡改 `STATUS_CHANGE` 记录；前端从不 POST `STATUS_CHANGE`。
- 引用已退役类型的历史问题正常显示。

**测试要求**：
- 单元：筛选参数拼接（含 `inspection_id`/`category`）；退役字典项标签渲染；`COMMENT`/`REASSIGN` 日志 payload（断言前端不构造 `STATUS_CHANGE`）。
- 组件：`ProblemEditForm` 下拉只列 active、列表渲染退役项；`ProblemProcessingLog` 只追加。
- E2E：筛选问题 → 进详情 → 改状态 → 处理记录新增一条 → 看全景。

## 7. 安全与合规要点

- **令牌**：access 短时 JWT 内存 + localStorage；refresh 不可解析（opaque，后端 Redis 可撤销）。登出必须调 `/auth/logout` 真正撤销。**绝不**打印 token / 密码。
- **前端鉴权仅 UI 隐藏，真正鉴权在后端 casbin**：菜单/按钮按 `/auth/me` 权限码隐藏，但所有受保护请求仍可能收 403（`error.code="FORBIDDEN"`）→ 必须处理，不可假设前端已挡住。
- **输入校验在边界**：所有表单 + 环境变量 + 后端返回的 jsonb 配置块用 zod 校验，fail-fast。
- **XSS**：问题 `description`/`note`、字典 `label` 等用户内容默认走 React 文本节点（自动转义），**禁止 `dangerouslySetInnerHTML`** 渲染用户内容。
- **CSRF**：使用 Bearer 头而非 cookie 会话，天然规避 CSRF；不要把 token 放可被跨站读取的 cookie。
- **媒体只读已确认**：只渲染 `verified_at` 非空（`CONFIRMED`）的媒体；签名 CDN URL 不缓存到持久存储。
- **坐标合规**：GCJ-02 仅在腾讯地图渲染边界存在，**绝不回传后端、绝不持久化**；后端永远 WGS84。
- **腾讯地图 key**：配置域名白名单（控制台），key 通过 `VITE_MAP_KEY` 注入，限制 referer。
- **ICP 备案（Day-1 运维强约束）**：中国大陆对外服务域名 **必须完成 ICP 备案**，否则无法上线；上线清单首项。
- **错误信息**：UI 给友好中文提示；详细堆栈仅控制台 / 上报，不向用户泄露后端内部细节。

## 9. 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| 后端 OpenAPI spec 未冻结，TS 客户端反复变 | 前端返工 | 以 spec 为契约，`npm run gen:api` 自动生成；spec 变更走评审；P0 先用 msw 按契约 mock 并行开发 |
| 腾讯地图 GCJ-02 偏移处理散落 / 双重转换 | 点位错位 | 强制唯一 `coordTransform`，单测 + ESLint 禁止他处实现；map 封装层统一坐标顺序 |
| SSE 被 CLB/CDN 缓冲导致进度卡住 | 导出体验差 | 必备轮询回退（同 job 对象）；Nginx 关 buffering |
| 真实 X3 导出图非 2:1 / 带黑边 | 全景渲染异常 | P5 提前用真机样本验证，反馈 App 调 `ExportMode.PANORAMA` 尺寸；PSV equirectangular 不依赖 GPano |
| 字典退役项导致历史数据渲染「未知」 | 数据可读性 | 字典缓存保留 `is_active=false` 项；`DictTag` 回退渲染 + 「已停用」标识 |
| 签名 CDN URL 过期 | 全景 / 下载失败 | 失败回调重拉 `GET /media/{id}` / 重查 job 取新 URL |
| 401 并发刷新风暴 | 体验抖动 / 死循环 | 单飞刷新合并；refresh 失败直接登出，refresh 请求跳过刷新拦截器 |
| 大量轨迹点 / 问题点地图卡顿 | 性能 | 用后端简化的 `route_geom`；MarkerCluster 聚合；必要时前端抽稀 |
| ICP 备案延误 | 无法上线（大陆） | Day-1 启动备案流程，列为上线清单首项阻塞项 |
| 前端误以为已鉴权而忽略 403 | 安全/体验 | 前端权限仅 UI 隐藏；所有请求显式处理 `FORBIDDEN` |

## 10. 性能预算 / 无障碍 / 响应式 / 视觉 UX 设计方向

> 本系统是「数据密集 + 地图 + 全景 WebGL」型管理后台。设计目标：高密度信息可读、地图/全景流畅、桌面优先并兼容平板，避免「默认模板感」。

### 10.1 性能预算（Core Web Vitals + Bundle）

| 指标 | 目标 | 说明 |
|---|---|---|
| LCP | < 2.5s | 登录页与列表首屏；列表骨架屏占位，避免大图阻塞 |
| INP | < 200ms | 表格筛选、地图交互、抽屉打开的响应延迟 |
| CLS | < 0.1 | 列表/卡片预留尺寸；图表与地图容器固定高度，避免回流抖动 |
| FCP | < 1.5s | 首屏关键 CSS 内联、非关键延后 |
| TBT | < 200ms | 重库（腾讯地图 GL、PSV、ECharts）按路由动态 `import()` |

**Bundle 预算（gzipped）**：管理后台主入口 JS < 300kb、CSS < 50kb。**重依赖必须按路由懒加载、不进主包**：

```ts
// 仅在进入对应路由时加载，不进首屏主包
const PanoramaViewer = lazy(() => import('@/shared/psv/PanoramaViewer'));   // PSV + three.js
const StatsOverviewPage = lazy(() => import('@/features/stats/StatsOverviewPage')); // ECharts
// 腾讯地图 GL SDK 通过 useTencentMap 在 P4 路由内 <script> 动态注入，绝不进主包
```

- **数据层**：TanStack Query 合理 `staleTime` + 分页（`meta.total/page/page_size`），避免拉全量；列表默认分页 ≤ 50 行。
- **地图**：用后端简化后的 `route_geom`；`MarkerCluster` 聚合大量问题点；轨迹点必要时前端抽稀。
- **全景**：`tier=web`（非 `original`）渲染；大图懒加载 + loading 占位；卸载 `viewer.destroy()` 释放 WebGL 上下文。
- **图表**：ECharts 卸载 `dispose()`，`ResizeObserver` 防错位；不在前端做大数据聚合（聚合在后端 D2）。
- **构建**：Vite `manualChunks` 拆分 `antd` / `echarts` / `psv` / `map` vendor chunk；产物 hash 文件名 + 长缓存。

### 10.2 无障碍（A11y）检查清单

- [ ] 语义化结构：`<header>/<nav aria-label="主导航">/<main>/<footer>`，不用裸 `div` 堆叠承担语义。
- [ ] 键盘可达：所有可操作元素 Tab 可达、有可见 `:focus-visible` 焦点环；模态/抽屉打开陷阱焦点（focus trap）、Esc 关闭、关闭后焦点归还触发元素。
- [ ] 表单：每个控件有 `<label>`/`aria-label`；校验错误用 `aria-invalid` + `aria-describedby` 关联 `error.details[].field`。
- [ ] 颜色对比：正文/标签文本对比度 ≥ 4.5:1；`dict_item.color` 标签若对比不足，自动叠加深色描边或前缀图标，**不以颜色作为唯一信息载体**（状态另带文字）。
- [ ] 地图/全景：纯指针交互组件提供文字替代入口（如「问题点列表」表格可达同等信息），地图非唯一查看路径。
- [ ] 动效：尊重 `prefers-reduced-motion`，减少地图飞行/图表入场动画。
- [ ] AntD `<ConfigProvider locale={zhCN}>` + 组件自带 ARIA；自定义组件补齐 `role`/`aria-*`。
- [ ] 自动化：CI 接入 `axe-core`（vitest/Playwright）对关键页做无障碍断言。

### 10.3 响应式断点

> 桌面优先（管理后台主战场），向下兼容平板；手机仅保证可读不保证全功能编辑。

| 断点 | 宽度 | 布局策略 |
|---|---|---|
| 移动 | 320 / 375 | 单列；表格转卡片或横向滚动；`ProLayout` 侧边收起为抽屉；只读优先 |
| 平板 | 768 | 侧边可收起；表格保留核心列，次要列入「更多」；地图/全景全宽 |
| 小桌面 | 1024 | 侧边固定 + 内容两栏；筛选区横排 |
| 标准桌面 | 1440 | 主工作分辨率；列表 + 详情抽屉并存 |
| 大屏 | 1920 | 最大内容宽度约束（避免行过长），统计页多卡片网格 |

- [ ] 各断点（320/375/768/1024/1440/1920）无横向溢出。
- [ ] 表格用 `ProTable` 列优先级 + 横向滚动；不在窄屏强塞全部列。
- [ ] 地图/全景容器用 `100%` 宽 + 固定高比，触摸手势可用（平板）。

### 10.4 视觉 / UX 设计方向（反模板）

**风格定位**：Swiss / 数据控制台（data-console）—— 克制、信息优先、强层级，而非「灰底白卡 + 一个点缀色」的默认模板。

- **设计令牌**：用 CSS 自定义属性集中定义调色板/字号/间距/动效时长，禁止散落硬编码。基于 AntD v5 的 `theme.token` + `cssVar` 输出，统一 `--color-*` / `--space-*` / `--radius-*` / `--duration-*`。
- **调色板**：中性灰阶为主骨架；**单一品牌主色**用于主操作与选中态；**语义色专用于状态**（问题状态 `dict_item.color` 走语义而非装饰），危险操作（删除）独立危险色。深浅主题二选一即可，不强制双主题；若做双主题两套都需有意设计。
- **排版层级**：通过字号对比建立层级（页面标题 / 区块标题 / 表头 / 正文 / 辅助说明），数字类（里程/时长/计数）用等宽数字（`font-variant-numeric: tabular-nums`）对齐。
- **节奏与留白**：列表/详情/统计区块间距有节奏，不是处处等距 padding；统计页用 bento/网格突出关键指标卡。
- **深度与状态**：抽屉/弹层/地图浮层用层级阴影区分；hover/focus/active/selected 都有刻意设计的状态（不只是默认描边）。
- **数据可视化纳入设计系统**：ECharts 主题与令牌一致（同一调色板/字体/网格线弱化）；图表是设计系统的一部分而非附属。
- **地图与全景作为一等公民**：地图浮层 InfoWindow、全景弹层标题区（项目/巡查/问题上下文）需精心排版，而非默认气泡。
- **空态/错误态/加载态**：统一 `EmptyState`/骨架屏/错误卡，文案中文友好、给出下一步操作，绝不空白或抛裸错。

**组件自检（落地前）**：是否避免了「默认 AntD 模板感」？hover/focus/active 是否刻意设计？是否用层级而非统一强调？是否像真实产品截图而非脚手架？
